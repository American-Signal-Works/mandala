---
title: "API Routes"
description: "Current route handlers used by the app and editor blocks."
---

The app primarily uses Server Components for reads and Server Actions for writes. API route handlers exist for collection and aggregate reads that are convenient for client-side hooks and editor blocks.

All current API routes require an authenticated Supabase user. Unauthenticated requests return `401`.

## GET /api/collections

Implemented in `apps/web/app/api/collections/route.ts`.

Returns visible collections for the current user and a compact field list for each collection.

Response shape:

```json
[
  {
    "id": "collection-id",
    "name": "Trades",
    "fields": [
      {
        "id": "field-id",
        "name": "Net P&L",
        "type": "currency",
        "collection_id": "collection-id"
      }
    ]
  }
]
```

## GET /api/collections/:id/fields

Implemented in `apps/web/app/api/collections/[id]/fields/route.ts`.

Returns all fields for a collection owned by the current user, ordered by `sort_index`.

## GET /api/collections/:id/rows

Implemented in `apps/web/app/api/collections/[id]/rows/route.ts`.

Returns rows for a collection owned by the current user.

Query parameters:

| Parameter | Default | Purpose                          |
| --------- | ------- | -------------------------------- |
| `limit`   | `100`   | Maximum number of rows to return |

## GET /api/collections/:id/rows/:rowId

Implemented in `apps/web/app/api/collections/[id]/rows/[rowId]/route.ts`.

Returns one row from a collection owned by the current user, or `null` if it does not exist.

## POST /api/aggregate

Implemented in `apps/web/app/api/aggregate/route.ts`.

Request body:

```json
{
  "collectionId": "collection-id",
  "metric": {
    "kind": "sum",
    "fieldId": "field-id"
  },
  "groupBy": ["field-id"]
}
```

Supported metric kinds:

- `count`
- `sum`
- `avg`
- `min`
- `max`

Ungrouped response:

```json
{
  "value": 123
}
```

Grouped response:

```json
{
  "groups": [
    {
      "key": {
        "field-id": "AAPL"
      },
      "value": 123
    }
  ]
}
```

## Error behavior

The aggregate route validates request bodies with Zod. Invalid input returns `400`. Runtime aggregate failures return `500` with an error message.

Collection read routes currently return empty arrays when no rows or fields are found.

## POST /api/mandala/workflows/fixtures

Runs one registered Mandala workflow fixture for an authorized company. The
response is private and is not cached.

The returned workflow event includes a bounded validation result:

```json
{
  "status": "blocked",
  "issues": [
    {
      "code": "source_data_stale",
      "message": "Source data is stale.",
      "kind": "reason"
    }
  ],
  "reasons": ["Source data is stale."],
  "warnings": [],
  "suppressRecommendation": true
}
```

Use `issues[].code` for automation, filtering, and routing. Codes are stable
identifiers; `message` is reader-facing wording and may change. `kind` is
either `reason` or `warning`.

The `reasons` and `warnings` arrays remain as compatibility projections for
existing consumers. New results derive those arrays from `issues`. Historical
message-only results remain readable and receive a safe
`legacy_unclassified_reason` or `legacy_unclassified_warning` code when
normalized.

The same validation object is stored in the workflow event and its
`event_validated` audit payload. The Mandala CLI fixture command renders only
the bounded status, suppression state, and code/message issue rows; it does not
render raw event, audit, source, or customer payloads.
