The app uses Supabase Auth with SSR cookie support from `@supabase/ssr`.

## Auth UI

The active auth routes are:

- `/login`
- `/sign-up`

`apps/web/components/auth/LoginAuthFlow.tsx` implements a three-step flow:

1. Email entry.
2. Link-sent state with resend support.
3. Success state after callback completion.

Email validation and normalization live in `apps/web/lib/auth/validation.ts`.

The Google and Microsoft buttons start Supabase OAuth in the browser:

- Google maps to Supabase provider `google`.
- Microsoft maps to Supabase provider `azure`.

The provider buttons require matching Supabase provider configuration before they can complete a live production sign-in.

Supabase OAuth does not expose the email OTP `shouldCreateUser` option. While sign-ups are enabled for the project, a first-time Google or Microsoft OAuth completion can create a Supabase user from either the sign-in or sign-up screen. If the product needs strict "sign in only" social auth, enforce that with Supabase Auth policy/hooks or an invitation/provisioning layer before enabling those providers broadly.

## Magic link request

The browser calls `requestEmailMagicLink()` from `apps/web/lib/auth/client.ts`.

That helper calls:

```ts
createClient().auth.signInWithOtp({
  email,
  options: {
    emailRedirectTo: getEmailRedirectTo(),
    shouldCreateUser,
  },
})
```

Sign-in mode passes `shouldCreateUser: false`. Sign-up mode passes `shouldCreateUser: true`.

`getEmailRedirectTo()` uses `NEXT_PUBLIC_SITE_URL` when set, otherwise it falls back to `window.location.origin`, and appends `/callback?next=%2Flogin%3Fauth%3Dsuccess`.

## Provider sign-on

The browser calls `requestOAuthSignIn()` from `apps/web/lib/auth/client.ts`.

That helper calls:

```ts
createClient().auth.signInWithOAuth({
  provider,
  options: {
    redirectTo: getAuthCallbackUrl(),
    ...(provider === "azure" ? { scopes: "email" } : {}),
  },
})
```

Microsoft sign-on uses Supabase's `azure` provider and requests the `email` scope because Supabase Auth requires Azure to return a valid email address.

Provider completion uses the same `/callback` route and then redirects to the auth success screen:

```text
/login?auth=success
```

### Provider setup

Production Google and Microsoft sign-on requires the external OAuth apps to be configured before Supabase can complete the flow.

For the Mandala Supabase project, add this callback URL to each external OAuth app:

```text
https://lbftopppjjdejvmwuxmq.supabase.co/auth/v1/callback
```

In Supabase Auth URL Configuration, the production allowlist must include:

```text
https://mandala.md/callback
https://www.mandala.md/callback
https://usebackdesk.com/callback
```

Local development should include the app origin used by `NEXT_PUBLIC_SITE_URL`, for example:

```text
http://localhost:3000/callback
```

Google Cloud setup:

- Create a Web OAuth client.
- Add `https://mandala.md` as an authorized JavaScript origin.
- Add the Supabase callback URL above as an authorized redirect URI.
- Enable the Supabase Google provider with the Google client ID and client secret.

Microsoft Entra setup:

- Register a Web application.
- Add the Supabase callback URL above as the Web redirect URI.
- Add a client secret and copy the secret value, not the secret ID.
- Enable the Supabase Azure provider with the application client ID and client secret.
- Keep the default common tenant URL unless the app should only allow one tenant or personal Microsoft accounts.
- Configure the optional `email` and `xms_edov` claims when using tenant settings that could emit unverified email domains.

## Callback completion

`/callback` is implemented by `apps/web/app/(auth)/callback/page.tsx`.

The callback page supports multiple Supabase callback shapes:

- `code` query parameter, completed with `exchangeCodeForSession()`
- `token_hash` query parameter, completed with `verifyOtp()`
- `access_token` and `refresh_token` hash parameters, completed with `setSession()`
- an already-present session from `getSession()`

On success, the route replaces the URL with the allowlisted `next` path. The default is:

```text
/login?auth=success
```

On failure, it replaces the URL with:

```text
/login?error=callback_failed
```

## Middleware

`apps/web/middleware.ts` refreshes the Supabase session and owns route-level redirects.

Protected routes are:

- `/`
- `/p/*`
- `/c/*`
- `/settings*`

Unauthenticated users on protected routes are redirected to `/login`.

Authenticated users visiting `/login` or `/sign-up` are redirected to `/`, except when the URL is `/login?auth=success`.

`/login?auth=success` is allowed to render even when middleware cannot yet observe the fresh Supabase session cookie. The callback page sets the browser session first, then routes to that success screen; stripping the query in middleware can make a valid magic-link callback look like a failed login.

## Server clients

Use the right Supabase client for the runtime:

| Runtime                              | File                               | Purpose                                              |
| ------------------------------------ | ---------------------------------- | ---------------------------------------------------- |
| Server Components and Server Actions | `apps/web/lib/supabase/server.ts`  | Authenticated Supabase client backed by Next cookies |
| Browser components                   | `apps/web/lib/supabase/browser.ts` | Browser Supabase client for auth interactions        |
| Admin-only server actions            | `apps/web/lib/supabase/admin.ts`   | Service-role client that bypasses RLS                |

The admin client must stay server-only.
